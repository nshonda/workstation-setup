# Research to Issues

Scan `_research/` for future work and create self-contained GitHub issues that serve as handoff documents.

## Preconditions

1. Check `_research/` exists at project root. If not: "No _research/ folder found in this project." — stop.
2. Detect repo from `git remote get-url origin` (Bash tool). Parse `owner/repo` from SSH or HTTPS URL. If no remote: "No git remote found." — stop.

## Phase 1: Scan for Candidates

Read all `.md` files in `_research/` (exclude `handoffs/` subdirectory). For each file, apply both detection methods:

### Method 1 — Frontmatter Status

Parse YAML frontmatter. Flag as candidate based on type + status:

| type | candidate statuses | skip statuses |
|------|-------------------|---------------|
| `spec` | `planning`, `active` | `completed`, `archived` |
| `plan` | `planning`, `queued` | `working`, `review`, `done` |
| `research` | skip (use method 2 only) | all |

Files without frontmatter or with unrecognized types: skip for method 1, still scan with method 2.

### Method 2 — Content Section Scanning

Within any file (regardless of frontmatter), look for section headings matching: "Future Work", "TODO", "Next Steps", "Backlog", "Follow-up". Also look for lines prefixed with `TODO:` or `FIXME:`.

Each qualifying section = one candidate. A section ends at the next heading of equal or greater depth. Bullets within a section become checklist items in the issue — do NOT split individual bullets into separate candidates.

### Candidate Structure

For each candidate, record:
- **title**: frontmatter `title` (method 1) or section heading text (method 2)
- **type**: `spec` | `plan` | `content-section`
- **content**: the relevant text to synthesize from

If zero candidates found: "No future work items found in _research/." — stop.

## Phase 2: Present and Select

Present a numbered list:

```
Found N future work items in _research/:

1. [spec] Calendar CBR Mirror Integration
2. [plan] Orchestration Layer - Phase 2
3. [content] Add rate limiting to API endpoints

Which items should become GitHub issues? (e.g., "1,3" or "all" or "none")
```

Wait for user selection. If "none": stop.

## Phase 3: Dedup Check

For each selected item ONLY (not all candidates), search existing GitHub issues via `search_issues` for title matches. If a likely match exists, report it:

```
Item 1 "Calendar CBR Mirror Integration":
  Possible duplicate: #38 "Calendar CBR Mirror Integration"
  Create anyway? (y/n)
```

Let user confirm or skip each flagged item.

## Phase 4: Create Issues

For each confirmed item, create a GitHub issue via `issue_write` with `method: create`.

### Issue Body Format

Synthesize from the source content. The issue is a **handoff document** — it must stand alone. A fresh Claude session or human should be able to pick it up and start working with zero additional context.

```markdown
## Background
[Why this work exists — synthesized from the research/spec context]

## Scope
[What needs to be done — clear boundaries]

## Key Decisions
[Decisions already made that constrain implementation]

## Checklist
- [ ] [Extracted from source, or synthesized from scope]

## Context
[Architecture notes, dependencies, gotchas]
```

### Rules for Issue Content

- **Synthesize, do not invent.** The issue body comes from what the `_research/` file says. Do not add implementation details, file paths, endpoint designs, or architectural decisions that aren't in the source material.
- **No back-references.** Never mention `_research/`, source filenames, local paths, or internal plan/spec numbering (e.g., "Plan 4", "Phase 2 of the design"). The issue stands alone — describe what exists by its function, not its origin.
- **No effort estimates.** Do not add complexity ratings, t-shirt sizes, or time estimates.
- **No attribution.** No "generated by", "created from", or similar footers.

### Labels

Apply if the label exists in the repo, skip silently if it doesn't:
- `spec` candidates: `enhancement`
- `plan` candidates: `task`
- `content-section` candidates: `enhancement`

### After Creation

Report each created issue: `Created: #42 "Calendar CBR Mirror Integration"`

## Common Mistakes

- Creating issues for ALL candidates without asking — always present the list and wait for selection
- Referencing `_research/` files in issue bodies — issues are handoff docs, not pointers
- Inventing implementation details beyond what the source material contains
- Running dedup search on every candidate upfront — only search selected items
- Treating each bullet in a "Future Work" section as a separate issue — bullets become checklist items within one issue
- Referencing internal plan numbers or phase names ("from Plan 4", "Phase 2") — describe what exists functionally instead
