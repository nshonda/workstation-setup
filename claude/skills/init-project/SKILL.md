# Init Project

Post-`/init` setup. Layers framework-aware skill references and project scaffolding on top of Claude's built-in initialization.

**Invocation:** `/init-project` or `/init-project --audit`

## Workflow

### 1. Detect Frameworks

Read dependency files and directory structure in the project root. Check one level deep for monorepos.

**package.json** (combine `dependencies` + `devDependencies`):

| Dependency | Skill(s) |
|------------|----------|
| `next` | `next-best-practices`, `vercel-react-best-practices` |
| `nuxt` or `@*/nuxt-layer-*` | `nuxt-dev` |
| `react` (without `next`) | `vercel-react-best-practices` |
| `vue` (without `nuxt`) | `vercel-react-best-practices` |
| `@supabase/supabase-js` | `supabase-postgres-best-practices` |

**composer.json** (combine `require` + `require-dev`):

| Dependency | Skill(s) |
|------------|----------|
| `silverstripe/*` | `ss-dev` |
| `roots/sage` or `roots/bedrock` or `wpackagist-*` | `wp-dev` |
| `laravel/framework` or `laravel/laravel` | `laravel-dev` |

**Directory structure:**

| Signal | Skill(s) |
|--------|----------|
| `wp-content/`, `wp-config.php`, or WP theme `style.css` | `wp-dev` |
| `nuxt.config.ts` or `nuxt.config.js` | `nuxt-dev` |
| `playbook.yml`, `site.yml`, or `roles/` + `inventory/` | `devops-infra` |

Deduplicate detected skills.

### 2. Scaffold _research/

- Create `_research/` directory if missing
- Ensure it's in `.gitignore` â€” add it if not present, create `.gitignore` if needed

### 3. Ensure CLAUDE.md Exists

If no `CLAUDE.md` exists, create a stub:

```markdown
# CLAUDE.md

This file provides guidance to Claude Code when working with code in this repository.
```

### 4. Inject Auto-Skills Block

If frameworks were detected, add or update the auto-skills section in CLAUDE.md using these markers:

```markdown
<!-- BEGIN auto-skills -->

## Framework Skills

Auto-detected by `init-project`. When working in this project, auto-invoke these skills:

- `skill-name`

<!-- END auto-skills -->
```

- If markers already exist, **replace** content between them (preserves manual content above/below)
- If no markers exist, **append** the block to the end of CLAUDE.md
- If no frameworks detected, skip this step

### 5. Create .claude/rules/framework.md

If frameworks were detected, create `.claude/rules/framework.md` (skip if it already exists):

```markdown
# Framework Rules

Auto-generated by `init-project`. Invoke these skills when working on matching files:

For [framework] files, invoke the `skill-name` skill.
```

Map each skill to its trigger context:

| Skill | Rule text |
|-------|-----------|
| `ss-dev` | For SilverStripe PHP files |
| `wp-dev` | For WordPress PHP files |
| `nuxt-dev` | For Nuxt/Vue files |
| `next-best-practices` | For Next.js files |
| `vercel-react-best-practices` | For React components |
| `supabase-postgres-best-practices` | For database queries and Supabase usage |
| `laravel-dev` | For Laravel PHP files |
| `devops-infra` | For infrastructure and deployment files |

### 6. Report

Print a summary: project name, detected frameworks, and what was created/updated.

## Audit Mode

When invoked with `--audit`, scan all git repos under `~/workstation/{personal,work}/`:

For each repo, check and report in a table:

| Check | Pass | Fail |
|-------|------|------|
| CLAUDE.md exists | has file | missing |
| `_research/` exists and gitignored | both | missing or not ignored |
| Auto-skills match detected frameworks | all listed | missing or stale entries |
| `.claude/rules/framework.md` exists (if frameworks detected) | has file | missing |

Print legend and summary count at the end.

After showing the table, ask: **"Want me to fix the issues?"** â€” then apply the init workflow to each repo that needs it.

## Important

- **Idempotent** â€” safe to run multiple times, updates in place
- **Don't touch manual CLAUDE.md content** â€” only modify between auto-skills markers
- **Don't duplicate skills** â€” deduplicate before writing
- **Check one level deep** for monorepo `package.json`/`composer.json` patterns
